# DO-Sync: Realtime Channel Sync Engine

A proof-of-concept realtime sync engine for a Slack-like channel using **TanStack DB** on the client and **Cloudflare Durable Objects + WebSockets** as the sync backend.

## Key Concepts

### `mutationId`
A UUID generated by the client for each mutation (create/update/delete). The server echoes it back in the `ack` and the canonical event. The client uses it to match pending optimistic mutations with server confirmations.

### `tempId`
A client-generated UUID used as the key for an optimistic insert. The server assigns a different canonical `id`. When the server broadcasts `message.created` with the matching `mutationId`, the client writes the canonical row (keyed by server id) to sync state and resolves the pending mutation promise, which causes TanStack DB to remove the optimistic overlay (keyed by tempId). React 18's automatic batching ensures no visible duplicate.

### `seq`
A monotonically increasing integer assigned by the ChannelDO to every mutation. Used for:
- **Ordering**: canonical message ordering across all clients
- **Replay cursor**: on reconnect, the client sends its `lastSeq` so the server can replay only missed events
- **Consistency**: clients know they're caught up when their `lastSeq` matches the server's current seq

### Snapshot
A full state dump sent when a client first connects (no `lastSeq`). The server sends:
1. `snapshot { seq, messages: [...] }` — all messages
2. `snapshot-end { seq }` — signals the end; client calls `markReady()`

### Replay
When a client reconnects with a `lastSeq`, the server checks its event log:
- If events after `lastSeq` exist: replay them as individual `message.created/updated/deleted` events
- Client applies each event to sync state, then receives `snapshot-end` to confirm catch-up

### `must-refetch`
If the event log has been pruned and doesn't cover the client's `lastSeq`, the server sends `must-refetch`. The client:
1. Resets `lastSeq` to 0
2. Re-subscribes without `lastSeq`
3. Receives a fresh full snapshot

## Protocol

### Client → Server

| Message | Fields | Purpose |
|---------|--------|---------|
| `subscribe` | `channelId, lastSeq?` | Join channel; get snapshot or replay |
| `message.create` | `mutationId, tempId, workspaceId, channelId, authorId, body` | Create message |
| `message.update` | `mutationId, id, body` | Update message body |
| `message.delete` | `mutationId, id` | Delete message |

### Server → Client

| Message | Fields | Purpose |
|---------|--------|---------|
| `snapshot` | `seq, messages[]` | Full state dump |
| `snapshot-end` | `seq` | End of snapshot/replay |
| `ack` | `mutationId, seq` | Quick mutation confirmation |
| `message.created` | `mutationId?, tempId?, seq, message` | Canonical create (broadcast) |
| `message.updated` | `mutationId?, seq, message` | Canonical update (broadcast) |
| `message.deleted` | `mutationId?, seq, id` | Canonical delete (broadcast) |
| `must-refetch` | — | Replay not possible |

## Optimistic Insert Flow

```
 Client                          Server (ChannelDO)
   │                                    │
   │ collection.insert({id: tempId})    │
   │ ──(optimistic row visible)──       │
   │                                    │
   │ ── message.create ──────────────►  │
   │    {mutationId, tempId, body}      │
   │                                    │ assigns canonical id + seq
   │                                    │ persists to SQLite
   │                                    │ appends to event_log
   │                                    │
   │  ◄──────────────── ack ──────────  │ {mutationId, seq}
   │                                    │
   │  ◄──────── message.created ──────  │ {mutationId, tempId, seq, message}
   │                                    │   (broadcast to ALL clients)
   │                                    │
   │ sync writes canonical row          │
   │ resolves pending mutation          │
   │ TanStack DB removes optimistic     │
   │ ──(canonical row visible)──        │
```

## Running Locally

### 1. Install dependencies

```bash
pnpm install
```

### 2. Start the worker (Durable Object backend)

```bash
cd apps/worker
pnpm dev
# Runs on http://localhost:8787
```

### 3. Start the web client

```bash
cd apps/web
pnpm dev
# Runs on http://localhost:3000
```

### 4. Test multi-client sync

Open `http://localhost:3000` in two browser tabs. Messages sent in one tab appear in the other in real time. Edit and delete operations also sync.

To test reconnect:
1. Stop the worker (`Ctrl+C`)
2. The client shows "disconnected"
3. Restart the worker
4. The client reconnects and replays missed events via `lastSeq`

## Browser SQLite Persistence

The collection uses `@tanstack/browser-db-sqlite-persistence` to persist data in an OPFS-backed SQLite database. This means:

- Messages survive page reloads
- Instant hydration on reload (data loads from local SQLite before the WebSocket connects)
- The sync adapter then replays missed events from the server
- Multi-tab coordination via `BrowserCollectionCoordinator` (BroadcastChannel + Web Locks)

## Design Decisions

- **Server-assigned IDs**: The server generates canonical message IDs (not client-generated). This demonstrates the full tempId → canonical ID reconciliation flow.
- **Event log for replay**: Every mutation is logged with its seq. Reconnecting clients replay from their `lastSeq` instead of downloading a full snapshot.
- **Event log pruning**: The log is bounded to 10,000 entries. If a client's `lastSeq` predates the oldest entry, `must-refetch` triggers a full snapshot.
- **No auth**: This is a sync engine POC. Authentication would be added at the worker level before the WebSocket upgrade.
- **Single entity**: Only `messages` to keep the POC focused. The pattern extends to any entity type.
